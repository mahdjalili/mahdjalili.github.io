در مورد Web Components کلی حرف و حدیث اشتباه وجود داره. خیلی از برنامه‌نویسا یا اصلاً نمی‌دونن به چه درد می‌خوره، یا یه تصور اشتباهی دارن از اینکه چی هست و چی نیست. این سری قراره اون شایعه‌ها و سوءتفاهم‌هایی که دور و بر این ابزارهای خیلی خفن ولی کم‌توجه‌شده وجود داره رو از بین ببره.

## یه نگاه به گذشته

اگه می‌خوای واقعاً Web Components رو قورت بدی، باید بدونی چی شد که اصلاً به وجود اومدن و چه فکری پشتش بوده.

### از بلبشو تا همکاری

ممکنه Web Components برات یه چیز تازه به‌نظر بیاد، ولی داستانش برمی‌گرده به چند دهه پیش. قبل از اینکه React، Angular یا حتی jQuery بیان وسط، یه دوره‌ای بود که اوضاع وب خیلی آشفته بود. خیلی‌ها اون دوران رو به «غرب وحشی وب» تشبیه می‌کنن، ولی به نظر من بیشتر شبیه دوران فئودالیسم بود، که هر مرورگر واسه خودش یه پادشاهی داشت!

تو دهه‌ی ۹۰ و اوایل ۲۰۰۰، مرورگرا با هم رقابت سنگینی داشتن: Netscape، Firebird، Firefox، Internet Explorer، Opera و بعدش Safari… هرکدوم راه خودشونو می‌رفتن و قابلیت‌های خاص خودشونو اضافه می‌کردن. برنامه‌نویسای بیچاره هم باید کلی ترفند می‌زدن تا سایتاشون تو همه مرورگرا درست کار کنه.

اون دوره به اسم «جنگ مرورگرها» معروفه. با اینکه W3C (اون گروهی که می‌خواست استاندارد تعریف کنه برای وب) تلاش می‌کرد همه‌چی رو سر و سامون بده، ولی خیلی کند پیش می‌رفت و همه مرورگرسازا هم از حرفش پیروی نمی‌کردن.

تا اینکه سال ۲۰۰۴ چندتا مهندس از Mozilla، Opera و Apple جمع شدن و یه گروه جدید زدن به اسم WHATWG. هدفشون این بود که ببینن مردم واقعاً چطوری از وب استفاده می‌کنن و بر اساس اون، پلتفرم وب رو توسعه بدن. نتیجه‌ش شد HTML5 و اولین جرقه‌های Web Components هم همون موقع زده شد.

اون موقع‌ها فیسبوک تازه یه پروژه دانشجویی بود و گوگل هم فقط GMail رو تازه معرفی کرده بود (فقط با دعوت می‌شد استفاده کرد)، و اصلاً کسی فکر نمی‌کرد بخواد مرورگر خودش رو بسازه. تا اینکه سال ۲۰۰۸ مرورگر Chrome رو داد بیرون و بازی عوض شد.

### ریشه‌های Shadow DOM

یکی از اولین نشونه‌های معماری به سبک کامپوننت‌ها، از سمت Mozilla اومد. رابط کاربری Netscape و بعدش Firefox با استفاده از دو تکنولوژی اختصاصی ساخته شده بود:

- XUL (زبان XML برای رابط کاربری)
- XBL (زبان اتصال XML)

با این ابزارها، مهندس‌ها می‌تونستن ویجت‌های قابل استفاده مجدد مثل دکمه‌ها، پنل‌ها، منوها و… بسازن، با رفتار دلخواه و استایل داخلی خودشون. به عبارتی، یه جورایی کامپوننت‌های اولیه بودن. ولی چون استاندارد نبودن، مرورگرهای دیگه هیچ‌وقت اونا رو استفاده نکردن.

تقریباً هم‌زمان که Mozilla داشت روی نسخه دوم XBL کار می‌کرد، مهندس‌های گوگل هم مشغول ساخت مرورگر Chrome بودن و با مشکلات مشابهی روبه‌رو شدن. مثلاً برای کنترل‌های پخش‌کننده‌ی `<video>`، نیاز داشتن ساختار داخلی المنت‌ها رو جوری بسازن که:

- به توسعه‌دهنده‌ها نمایش داده نشن،
- و استایل‌های صفحه روی اون‌ها تأثیر نذاره.

راه‌حلشون شد Shadow DOM؛ یه مکانیزم بومی مرورگر برای کپسوله‌سازی واقعی. اولش فقط برای ویجت‌های داخلی مرورگر استفاده شد، ولی اون‌قدر ابزار قدرتمندی بود که بعدها پتانسیل استفاده گسترده‌تری براش در نظر گرفتن.

### از ابزار داخلی تا قدرت‌دادن به توسعه‌دهنده‌ها

اون چیزی که امروز به اسم Web Components می‌شناسیم، از وقتی شکل جدی گرفت که این قابلیت‌های سطح پایین مرورگر به‌صورت API قابل استفاده برای توسعه‌دهنده‌ها پیشنهاد شد.

سال ۲۰۱۱، الکس راسل توی کنفرانس Fronteers یه ارائه تاریخی داشت. اون نشون داد تیم Chrome چطور داره تلاش می‌کنه قدرت Shadow DOM رو در اختیار توسعه‌دهنده‌های فرانت‌اند بذاره.

از اون‌جا بود که کار روی استانداردسازی این فناوری‌ها با همکاری WHATWG و W3C شروع شد و شرکت‌های سازنده مرورگرها دست به دست هم دادن تا پیاده‌سازی‌هایی سازگار، قابل اطمینان و قابل همکاری ارائه کنن.

## حالا چه خبره؟

حالا که فهمیدیم Web Components از کجا اومدن، وقتشه ببینیم چی شد که به وضعیت امروزی رسیدن.

### اوج‌گیری فریم‌ورک‌های مدرن

الان دیگه Web Components توی همه مرورگرهای مدرن پشتیبانی می‌شن. ولی این اتفاق یه‌شبه نیفتاد. تو همین مدت، دنیا داشت جلو می‌ رفت و شرکت‌ها دنبال راه‌های قابل اعتماد برای ماژولار کردن اپلیکیشن‌هاشون بودن. تو این مسیر، کلی فریم‌ورک اومدن و رفتن — بعضیا محبوب شدن، بعضیا فراموش شدن.

هر کدوم از این فریم‌ورک‌ها سعی داشتن جاهایی رو که Web APIها پوشش نمی‌دادن، پر کنن.

در حالی که خیلی از این فریم‌ورک‌ها طرفدارای پر و پا قرص پیدا کردن، یه فریم‌ورک بیشتر از همه خودشو نشون داد: React.

محبوبیت React انکارناپذیره، ولی مشکل اینجاست که این ابزار کاملاً موازی با پلتفرم اصلی وب توسعه پیدا کرده؛ یعنی دنیای خودش، اکوسیستم خودش و حتی انتزاع‌های خودش رو ساخته که هیچ‌وقت با مسیر استانداردهای وب هم‌راستا نبوده.

این روند یادآور همون دوران جنگ مرورگرهاست — فقط با یه تفاوت مهم: Meta (فیسبوک سابق) مرورگر نمی‌سازه.

### دو قدم جلو، یه قدم عقب

React کلی ویژگی داره که در نگاه اول شبیه چیزیه که Web Components ارائه می‌ده. ولی این ویژگی‌ها به چیزهایی مثل JSX متکی‌ان — که یه اکستنشن غیراستاندارد محسوب می‌شه. این قابلیت‌ها درواقع یه لایه‌ی اضافه روی پلتفرم وب هستن و خیلی وقتا بر خلاف مسیری هستن که گروه‌هایی مثل WHATWG برای آینده وب در نظر گرفتن.

آخرین باری که یه تکنولوژی اختصاصی در برابر استانداردها ایستاد، نتیجه‌اش شد Flash. خیلی از نسل جدید احتمالاً حتی اسمش رو هم نشنیدن، ولی ماهایی که اون دوران رو دیدیم، یادمونه که مجبور شدیم نصف وب رو از اول بسازیم تا از شر Flash خلاص بشیم.

الان دیگه تقریباً هیچ اثری از Flash توی وب باقی نمونده — با اینکه فقط چند ساله رسماً کنار گذاشته شده.

### پایداری نتیجه داد

مسیر رسیدن Web Components به وضعیت فعلی، اصلاً راحت نبود. کلی پیشنهاد مطرح شد، تست شد، رد شد، دوباره بازطراحی شد تا بالاخره به این نقطه رسیدیم. اون چیزی که حالا تو همه‌ی مرورگرها و دستگاه‌های مدرن در دسترسه، سال‌ها زمان برد تا درست و حسابی شکل بگیره.

برای ماهایی که از روز اول دنبالش بودیم، این یه لحظه‌ی خوشحال‌کننده‌ست — چون بالاخره رؤیاهامون به واقعیت تبدیل شدن.
Shadow DOM و APIهای مرتبط باهاش یه دنیای کاملاً جدید از طراحی کامپوننت‌ها رو باز کردن که تازه داره پتانسیل واقعیش رو نشون می‌ده.

## آینده

Web Components بالاخره به بلوغ رسیدن، ولی داستان اصلی تازه داره شروع می‌شه. حالا که APIهای پایه‌ای به‌طور گسترده پشتیبانی می‌شن، وارد یه فاز جدید شدیم — نه فقط فاز استفاده، بلکه فاز نوآوری.

### پشتیبانی از استانداردها

الان موجی از فریم‌ورک‌ها، کتابخونه‌ها و ابزارهای سبک‌وزن به‌وجود اومده که بر پایه‌ی Web Components ساخته شدن تا تجربه‌ی برنامه‌نویسی رو بهتر کنن. بین اینا، دو تا پروژه حسابی خودی نشون دادن:

- Lit از گوگل: یه کتابخونه‌ی مینیمال که مفاهیمش شبیه React هست. حتی کتابخونه‌های کامپوننتی مثل Shoelace که روی Lit ساخته شدن هم حسابی دارن معروف می‌شن.
- FAST از مایکروسافت: یه لایه‌ی نازک از انتزاع که پایه‌ی کتابخونه‌ی Fluent UI رو تشکیل می‌ده — مجموعه‌ای از کامپوننت‌های تم‌پذیر و قابل دسترس برای توسعه‌ی UI در سطح سازمانی.

برخلاف فریم‌ورک‌های قدیمی که ساختار خودشون رو به‌وجود می‌آوردن، Lit و FAST کاملاً از پلتفرم وب پشتیبانی می‌کنن و Web Components رو به شکلی هماهنگ با استانداردها گسترش می‌دن.

### غول‌ها وارد می‌شوند

شرکت‌های بزرگ مثل GitHub، Reddit، Salesforce، Adobe و خیلی‌های دیگه یا کاملاً به معماری‌های مبتنی بر Web Components مهاجرت کردن یا در حال حرکت به اون سمت هستن.

دلیلش هم صرفاً هیجان یا مد روز بودن نیست. جریان داره تغییر می‌کنه چون Web Components سه چیز مهم رو به‌صورت تضمینی ارائه می‌ده:
ماندگاری بالا، سازگاری بین مرورگرها، و عملکرد خوب.
همون چیزهایی که React و بقیه‌ی فریم‌ورک‌ها اول قولش رو داده بودن — ولی حالا بدون نیاز به لایه‌های اضافی و انتزاع‌های شخص ثالث، در دسترسه.

### فقط یه تیکه از پازل

Web Components یه راه‌حل همه‌کاره نیستن. به‌تنهایی نمی‌تونن همه‌ی امکاناتی که یه فریم‌ورک کامل می‌ده رو تأمین کنن.
اونا فقط یه بخش از مجموعه‌ی بزرگ‌تری از ابزارها هستن که داره به سمت ساخت اپلیکیشن‌های وب قوی‌تر و پیشرفته‌تر پیش می‌ره.

امروزه اپ‌های وب امکانات و انعطافی دارن که قبلاً فقط مخصوص اپ‌های بومی (native) بودن.

بیاین یه نگاهی بندازیم به چند تا استاندارد نوظهور که قراره نسل بعدی توسعه وب رو شکل بدن:

- Declarative Shadow DOM: رندر سمت سرور بدون نیاز به hydration.
- WebGPU: پردازش و رندرینگ با عملکرد بالا داخل مرورگر.
- WebUSB و WebSerial: دسترسی مستقیم به دستگاه‌های متصل.
- WebSocketStream: استریم داده‌ی بلادرنگ به‌صورت بهینه.
- Sanitizer API: تزریق امن محتوای تولیدشده توسط کاربر.
- View Transitions: انیمیشن برای تغییر وضعیت DOM.

هر کدوم از اینا به‌تنهایی یه جهش بزرگن. ولی وقتی کنار هم قرار می‌گیرن، نشون می‌دن که داریم وارد یه دوران پسافریم‌ورک می‌شیم — جایی که خود پلتفرم وب اون‌قدر قدرتمند شده که بدون نیاز به واسطه، می‌شه باهاش اپ‌های مدرن، امن، و بهینه ساخت.
همه‌ی این قابلیت‌های جدید، فقط ارتقا هستن — زیرساخت آماده‌ست، وقت ساختنه!

### پلتفرم پابرجاست

Web Components نه یه مد زودگذرن، نه یه ترند لحظه‌ای. اونا ادامه‌ی طبیعی چیزیه که همیشه باعث موفقیت وب شده:
باز بودن، ماژولار بودن، و ارتقای تدریجی.

مسئله فقط این نیست که چی کار می‌کنن، بلکه اینکه چطوری بی‌دردسر و طبیعی توی ساختار وب جا می‌گیرن، طوری که انگار از اول باید اونجا می‌بودن.

با پایه‌هایی مثل Web Components، ماژول‌های ES و کلی API بالغ یا در حال رشد، آینده‌ی وب واقعاً روشنه.
قبلاً مجبور بودیم با فریم‌ورک‌ها جاهای خالی رو پر کنیم.
اما امروز اون شکاف‌ها داره یکی‌یکی پر می‌شن.
و شاید فردا به دوران فریم‌ورک‌ها هم همون‌طور نگاه کنیم که حالا به Flash نگاه می‌کنیم: یه ماجراجویی عجیب توی مرزهای ناشناخته‌ی وب — که فقط اونا که اونجا بودن، واقعاً می‌تونن درکش کنن.
