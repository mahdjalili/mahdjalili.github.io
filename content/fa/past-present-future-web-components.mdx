دور و بر Web Components کلی شایعه و سوءتفاهمه. خیلی از دولوپرها هنوز نمی‌دونن فایده‌شون چیه یا فکر می‌کنن نمی‌شه باهاشون کاری کرد. این سری اومده همین حرفای بی‌پایه رو جمع کنه و حق این ابزار کم‌شاخ‌و‌برگ رو ادا کنه.

## گذشته

برای این‌که واقعاً مسلط شی، باید بدونی چرا و چطور اصلاً Web Components سر و کلّه‌شون پیدا شد.

### از هرج‌ومرج تا همکاری

شاید Web Components جدید به نظر برسن، ولی داستان‌شون چند دهه عقب‌تر می‌ره. قبل از React و Angular و حتی jQuery، یه دوران وحشی بود؛ بعضیا می‌گن Wild West، من می‌گم عصر فئودالی! هر مرورگر واسه خودش یه قلمرو بود.

اواخر دهه ۹۰ و اوایل ۲۰۰۰، Netscape/Firebird/Firefox، اینترنت اکسپلورر، اوپرا و بعداً سافاری تو یه رقابت تنگاتنگ بودن. هرکدوم فیچر اختصاصی می‌زد، دولوپرها هم مجبور بودن کلی ترفند بزنن که سایت یه‌دست کار کنه.

اسم این دوران شد «جنگ مرورگرها». W3C می‌خواست HTML و CSS رو استاندارد کنه، ولی پیشرفت لاک‌پشتی بود و همه هم حرف‌شنو نبودن.

نتیجه؟ مهندس‌های موزیلا، اوپرا و اپل در ۲۰۰۴ یه گروه جدید ساختن: **WHATWG**. هدفشون این بود که وب رو طبق نیاز واقعی کاربرا جلو ببرن. خروجیش HTML5 شد و بذر Web Components هم همون موقع کاشته شد.

اون موقع فیس‌بوک تازه یه پروژه دانشگاهی بود و گوگل داشت جیمیل رو فقط با دعوتنامه می‌داد. کروم؟ سال ۲۰۰۸ اومد.

### تولد Shadow DOM

اولین جرقه‌های معماری کامپوننتی از موزیلا شروع شد. UI نت‌اسکیپ و فایرفاکس روی دوتا تکنولوژی اختصاصی می‌چرخید:

- **XUL** (زبان XML برای رابط کاربری)
- **XBL** (زبان XML برای بایند کردن)

مهندس‌ها با اینا ویجت‌های قابل‌استفاده‌مجدد می‌ساختن. ولی استاندارد نبودن، مرورگرای دیگه هم محل نذاشتن.

همون موقع که موزیلا روی XBL2 کار می‌کرد، گوگل برای کروم به مشکل مشابه خورد: مثلاً کنترل‌های `<video>` رو می‌خواستن جوری بسازن که نه استایل صفحه خرابش کنه نه دست دولوپر به دل و روده‌ش برسه. جوابشون شد **Shadow DOM**؛ یه حریم خصوصی واقعی برای المان‌ها. اول فقط توی خود مرورگر استفاده می‌شد، ولی جذابیتش باعث شد ایده‌ی بزرگ‌تری پشتش شکل بگیره.

### از ابزار داخلی تا قدرت دست دولوپر

Web Componentsِ امروزی وقتی جدی شد که همین قابلیت‌های low-level قرار شد برسن دست فرانت‌اندکارا. ۲۰۱۱، الکس راسل تو کنفرانس Fronteers یه دمو تاریخی داد و نشون داد تیم کروم چطور می‌خواد Shadow DOM رو عمومی کنه. از همون موقع WHATWG و W3C دست به کار استانداردسازی شدن و مرورگرها نشستن پای یه پیاده‌سازی مشترک.

## حال

حالا که فهمیدی از کجا شروع شد، بریم ببینیم چطور به وضعیت فعلی رسید.

### اوج گرفتن فریمورک‌های مدرن

الان Web Components تو همه مرورگرای به‌روز ساپورت می‌شن، ولی یه شبه این‌طور نشد. کسب‌وکارها نیاز داشتن اپ‌هاشونو ماژولار کنن، پس کلی فریمورک اومد و رفت. هرکدوم می‌خواست حفره‌های Web APIها رو پر کنه.

تو این شلوغی، React مثل یه غول ظاهر شد. محبوبیتش غیرقابل انکاره، ولی مسیرش از پلتفرم اصلی جدا بود و برای خودش اکوسیستم ساخت؛ دقیقاً مثل جنگ مرورگرها، فقط این بار متا اصلاً مرورگرساز نیست.

### دو قدم جلو، یه قدم عقب

React یه‌سری قابلیت داره که ظاهرش شبیه Web Componentsه، ولی رو افزونه‌های غیر استاندارد مثل JSX می‌چرخه. یعنی لایه‌ایه روی پلتفرم، نه تو دلش. دفعه قبل که تکنولوژی اختصاصی اومد وسط نتیجه‌اش شد فلش. اگه سنت قد بده، یادته چه‌جوری نصف وبو دوباره کدنویسی کردیم که فلش بره کنار.

### پایداری جواب می‌ده

راهی که Web Components رو به امروز رسوند صاف و صوف نبود. کلی پیشنهاد رد و بدل شد تا بالاخره نسخه قابل اتکا دراومد. حالا Shadow DOM و رفقاش یه دنیای تازهٔ کامپوننت‌سازی رو باز کردن و هنوز اول راهیم.

## آینده

Web Components بالاخره بالغ شدن، ولی داستان اصلی تازه شروع شده. با ساپورت سراسری، وارد فاز نوآوری شدیم؛ نه فقط استفاده، بلکه ارتقا.

### احترام به استاندارد

یه موج لایب و فریمورکِ سبک درحال بالا اومدن هست که روی Web Components سوار می‌شن تا تجربه کدنویسی رو بهتر کنن. دوتا ستاره‌ش:

- **Lit** از گوگل: یه لایبرری مینیمال با حال و هوایی شبیه React. کتابخونه‌های کامپوننتی مثل Shoelace هم روی Lit پرواز کردن.
- **FAST** از مایکروسافت: لایه نازکیه که پایه Fluent UI محسوب می‌شه؛ مجموعه‌ای از کامپوننت‌های تم‌پذیر و دسترس‌پذیر برای UI سازمانی.

فرقشون با فریمورکای قدیمی اینه که به‌جای ساخت دنیاهای موازی، دست رو دست پلتفرم می‌ذارن.

### اسم‌های بزرگ، تغییرهای بزرگ

کمپانی‌های گنده مثل گیت‌هاب، ردیت، سیلزفورس، ادوبی و کلی‌های دیگه یا رفتن سمت Web Components یا دارن می‌رن. حرکت داره عوض می‌شه؛ نه بخاطر هایپ، بخاطر دوام، سازگاری و کارایی ــ همون چیزایی که React قولش رو داده بود، منتها الان بدون کتابخونه سنگین.

### یه تکه از پازل

Web Components عصای موسی نیست. تنهایی همه کار یه فریمورک رو نمی‌کنه. فقط یه بخش از ابزارهای جدیده که دارن وب اپ‌ها رو قوی‌تر می‌کنن. امروز وب کارایی می‌کنه که یه زمانی فقط اپ‌های نیتیو بلد بودن.

چندتا استاندارد درحال ظهور که آینده رو می‌سازن:

- Declarative Shadow DOM: SSR بدون هایدریشن.
- WebGPU: رندر و محاسبات سنگین تو مرورگر.
- WebUSB و WebSerial: دسترسی مستقیم به دیوایس‌ها.
- WebSocketStream: استریم زنده و به‌صرفه.
- Sanitizer API: تزریق امن محتوای کاربر.
- View Transitions: انیمیشن تغییر وضعیت DOM.

تک‌تک‌شون یه جهش رو به جلوئن؛ کنار هم یعنی عصر پسافریمورک نزدیکه؛ جایی که پلتفرم وب خودش اون‌قدر قویه که با کمترین ابزار، اپ‌های مدرن رو بسازی.

### پلتفرم می‌مونه

Web Components مُد نیست؛ ادامه منطقی همون چیزیه که وب رو خاص کرد: باز بودن، ماژولار بودن و ارتقای تدریجی. مسئله فقط کارایی نیست، اینه که چقد طبیعی تو تار و پود وب می‌شینن.

با Web Components، ES Modules و یه عالمه API بالغ و تازه‌نفس، آینده وب روشنه. یه زمانی واسه پر کردن خلأها به فریمورک نیاز بود؛ امروز اون خلأها دارن پر می‌شن. فردا شاید به فریمورک‌ها مثل فلش نگاه کنیم: یه ماجراجویی هیجان‌انگیز که باید توش می‌بودیم تا بفهمیم چی گذشت!
